Multi-Agent Workflow Design

The system implements a five-role multi-agent pipeline for each ADD iteration. Each agent receives a dedicated system prompt with role-specific instructions, dialogue rules, and the shared prior knowledge bundle. The agents collaborate sequentially within each iteration:

#Agent RoleResponsibilityADD Steps Covered
1Requirements AnalystSelects iteration drivers from the prior knowledge, frames the design scope, identifies candidate design concepts and ambiguitiesStep 2, Step 3
2Solution ArchitectProposes the architectural design: design concepts, element instantiation, interface definitions, and design decisionsStep 4, Step 5, Step 6
3Quality ReviewerReviews the proposal for compliance with selected drivers and assignment constraints; flags violations and requests revisionsCollaborative verification
4Solution Architect (revision)Revises the design based on reviewer feedback, producing the final architectural outputStep 4, Step 5, Step 6 (revised)
5Diagram CuratorConverts the approved architecture into a self-contained Mermaid diagramStep 6 (view)
6Iteration ModeratorConsolidates all agent outputs into a structured JSON result covering Steps 2–7Step 7 + consolidation
Key Design Principles
  • No external domain knowledge: all agents are constrained to use only the provided prior knowledge bundle (ADD 3.0 method, Hotel Pricing System case study, and the fixed iteration plan).
  • No few-shot examples: no demonstration outputs are included in any prompt.
  • Traceable decisions: every design decision must reference specific driver IDs (e.g., QA-3, CON-6) from the prior knowledge.
  • Views in Mermaid code: all architectural views are produced as renderable Mermaid flowchart diagrams.

Iteration 1: Establishing an Overall System Structure

ADD Step 2 — Establish Iteration Goal by Selecting Drivers

The selected drivers for this iteration were:

CRN-1 CON-6 CON-2 CON-5 QA-5 QA-3 QA-4

Iteration goal: Define the system context, identify major architectural containers (services or components), allocate high-level responsibilities among them, and specify initial interfaces — especially those related to user authentication, price querying, and integration with external systems — while adhering to cloud-native principles and REST-based interoperability.

These drivers shape the system boundaries (CRN-1), deployment model (CON-6), cloud identity integration (CON-2), REST-first interfaces (CON-5), security enforcement (QA-5), query availability (QA-3 at 99.9%), and scalability (QA-4 up to 1M queries/day).

ADD Step 3 — Choose Elements to Refine

The elements refined in this iteration were:

  • The system boundary with external actors: users (via browser), the User Identity Service, and the Channel Management System
  • Core internal containers realizing HPS-1 through HPS-6
  • Primary inbound/outbound interfaces: UI, query API, management APIs, and the price publication channel
  • Initial cloud-native deployment assumptions

ADD Step 4 — Choose Design Concepts

The chosen design concepts were:

  • Cloud-native microservices architecture — functional decomposition into independently deployable services to satisfy CON-6 and enable modifiability
  • API Gateway pattern — a single public entry point for browser and API traffic, centralizing security and protocol handling
  • Read-write separation — separating price query (read) from price update (write) paths to optimize performance (QA-4) and availability (QA-3)
  • Externalized identity — delegating authentication to the cloud provider identity service per CON-2
  • Containerized stateless deployment units — enabling horizontal scaling and CI/CD readiness

ADD Step 6 — Sketch Views and Record Design Decisions

Key decisions:

  1. Use separate services for hotel/rate management, price update, and price query responsibilities — enabling independent scaling and modifiability
  2. Place browser and external API access behind the API Gateway — centralizing security and protocol handling
  3. Keep identity validation outside the Hotel Pricing System — use authorization context (user-to-hotel mappings) inside backend services
  4. Separate price query from price update responsibilities so the query path can be scaled and kept available independently
  5. Keep external integrations behind explicit interfaces so REST can be used first and future protocol additions remain possible (CON-5)
  6. Containerize all services as stateless units for cloud deployability and CI/CD readiness (CON-6)

3. Individual Reflection

3.1 Problems Encountered and Solutions Adopted

#ProblemSolution Adopted
1 External knowledge leakage: Some model outputs used named technologies or pattern labels (e.g., specific protocol names, design pattern labels) that were more specific than what appears in the assignment's prior knowledge bundle. The final report was reviewed and conservative wording was used so that design decisions remained grounded in the provided drivers, constraints, and concerns. The prompt's dialogue rules were also strengthened to instruct agents to describe responsibilities generically when a named pattern is absent from the prior knowledge.
2 Invalid Mermaid syntax: The original generated diagram output included component-diagram syntax that was not standard Mermaid graph syntax and would not render correctly in standard tools. The final submission version preserves the same architectural content but rewrites all diagrams as valid flowchart TD or flowchart LR diagrams that render correctly in Mermaid.
3 Cross-iteration consistency: Four ADD iterations can become architecturally inconsistent if each round is treated independently, since later decisions may contradict or ignore earlier ones. The workflow passed prior iteration summaries (including key decisions, selected drivers, and residual risks) into later agent turns so that each round refined the previous decisions instead of restarting the design from scratch.
4 Auditability of multi-agent output: The multi-agent process produced many intermediate responses (24 agent turns total), making it difficult to verify that the final consolidated result accurately represents the discussion. The application archived full conversation logs (with timestamps, system prompts, user prompts, and model responses for every turn), structured iteration JSON snapshots, and a generated report so the result can be audited against the actual run.
5 Driver pre-selection bias: Early versions of the code included suggested driver lists in the prompt, which could bias the model toward pre-selected drivers rather than independently selecting from the prior knowledge. The suggested driver candidates were removed from the analyst prompt. The agent now independently selects iteration drivers grounded solely in the prior knowledge bundle, which better demonstrates the model's ability to reason about architectural concerns.
